# コメント規約 (和文スタイル)

目的: 本プロジェクトで使用するコメントの書き方を統一し、可読性・検索性・自動ドキュメント生成に寄与する。

著者表記: `淵脇 未来.`
日付書式: `YYYY/MM/DD`（例: `2025/10/05`）
キーワード: `TODO` / `FIXME` / `NOTE` は大文字で統一。必要ならチケット番号を併記する。

---

## 1. ファイルヘッダ（ヘッダ／ソースファイル先頭）
用途: ファイルの目的を短く説明する。必ず追加する。

テンプレート:
```
/**********************************************************************************
* @author    : 淵脇 未来.
* @date      : 2025/10/05.
* @brief     : ファイルの目的を短く(1〜2行).
**********************************************************************************/
```

例:
```
/**********************************************************************************
* @author    : 淵脇 未来.
* @date      : 2026/01/02.
* @brief     : 仮想パッド入力ラッパー.
             : 複数入力ソースをゲームアクションへマッピングする.
**********************************************************************************/
```

---

## 2. クラス／struct コメント

クラス（ヘッダ内、クラス定義の直前）:
```
/**********************************************************************************
* @brief : クラスの役割と責務を短く(1〜2行).
* @note  : 特記事項（所有権、スレッド制約、依存関係など）があれば追記.
**********************************************************************************/
```

構造体（短い場合は行コメントでも可）:
```cpp
// 役割を短く書く.
struct Foo { ... };
```

---

## 3. public getter / setter

Getter:
```cpp
// ~~を取得.
Type GetX() const noexcept;
```

Setter:
```cpp
// ~~を設定.
void SetX(Type v) noexcept;
```

---

## 4. 公開関数（詳細コメント）
用途: 処理の概要、引数、戻り値、必要なら例外条件を記載する。

テンプレート:
```
/**********************************************************
* @brief                : 処理の概要(1行).
* @param[in] paramName  : 説明(入力).
* @param[out] outName   : 説明(出力).
* @return               : 処理結果の意味.
**********************************************************/
```

注意: 引数が多い場合は箇条書きで項目を分ける。

---

## 5. メンバ変数
行コメントで短く記述。
例:
```cpp
float m_HP; // プレイヤーの現在HP.
```

命名規約は別ファイル（`CodingStyle_Naming.md`）で管理することを推奨。

---

## 6. 列挙値
各列挙値に短い説明を付ける。例:
```cpp
enum class eState { 
    Idle, // 待機状態.
    Run   // 走行中.
};
```

---

## 7. TODO / FIXME / NOTE
- 全て大文字で記述。
- 可能ならチケット番号や短い説明を追加。
例: `TODO(JIRA-123): 入力バッファの閾値を調整する`。

---

## 8. 自動化／運用
- CI に Doxygen を導入する場合は、`@brief` や `@param` コメントのテンプレを満たすこと。
- pre-commit フックでファイルヘッダと公開関数の `@brief` 存在チェックを行うと効果的。
- 新規ファイルは必須、既存コードは段階的に適用する（PR にてコメント追加を必須にする）。

---

## 9. 例: 完全な関数コメント
```
/**********************************************************
* @brief            : 指定アクションが押された瞬間か判定する.
* @param[in] action : 判定するゲームアクション.
* @return           : 押された瞬間なら true.
**********************************************************/
bool IsActionPress(eGameAction action) const;
```
